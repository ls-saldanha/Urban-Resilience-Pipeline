
### 1. Ingestion (Python + Cloud Run)
* **Multi-Source Extraction:** A Python script connects to the **OpenAQ API** (Air Quality) and **Overpass API** (Infrastructure) to collect real-time environmental data.
* **Deployment:** The ingestion service is containerized using **Docker** and deployed on **Cloud Run**, providing an elastic, serverless execution environment.
* **Landing Zone:** Raw JSON responses are stored directly in **GCS** under `/landing/air_quality/` and `/landing/infrastructure/`.



### 2. Geospatial Processing (PySpark + Dataproc)
* **Parsing Complexity:** A **PySpark** job handles the complex, nested JSON structures from the APIs, flattening them into a tabular format.
* **Spatial Mapping:** The job uses Spark to correlate geographic coordinates (latitude/longitude) from air sensors with specific urban zones or neighborhood boundaries.
* **Refining:** Data is partitioned by `date` and `region` and saved as Parquet files in the `/silver/` storage layer.

### 3. Analytical Modeling (BigQuery + dbt)
* **Staging:** Tables are created in **BigQuery** using the Parquet files as a source, maintaining a clear lineage from the raw data.
* **Index Creation:** Use **dbt** to create the final "Urban Resilience Index":
    * **Aggregation:** Calculate daily averages of pollutants ($PM_{2.5}$, $NO_2$) per neighborhood.
    * **Correlation:** Join pollutant averages with the density of public transport points (bus stops, subway stations) to measure environmental efficiency.
* **Materialization:** Final models are materialized as tables in the Gold layer for high-speed dashboard consumption.

### 4. Idempotency & Quality Assurance
* **Merge Logic:** The pipeline uses a `MERGE` SQL strategy in dbt to ensure that re-running the process for the same day updates existing records rather than duplicating them.
* **Geospatial Validation:** Automated checks verify that all coordinates fall within the expected city boundaries before the data reaches the final dashboard.
